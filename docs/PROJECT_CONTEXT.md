# 项目背景与上下文

## 项目是什么
`GobyShrimp` 是一个面向港股市场的、由 `LLM（大语言模型）` 与确定性执行层共同驱动的可审计策略工厂。

它不是当前就追求自动实盘的交易机器人，而是一个先完成：
- 研究
- 回测准入
- 模拟盘
- 审计
- 实盘准入证据

的长期运行系统。

## 当前阶段定义
当前项目处于：
- `v1` 基本完成
- `v2` 已启动并持续推进

阶段目标：
- `v1`：让研究、治理、模拟盘、审计闭环成立
- `v2`：证明系统长期稳定，形成明确实盘准入证据
- `v3`：在小资金、强风控、可回滚前提下进入受控实盘

当前明确边界：
- 允许自动进入 `paper trading（模拟盘）`
- 不允许自动进入 `live trading（实盘）`
- 实盘必须满足独立的准入标准，并保留人工确认

## 最终目标
最终目标不是“做一个能展示研究结果的 dashboard”，而是：
- 在金融市场中实现可持续的真实收益

当前工程路线是：
- 先把系统做成“值得上实盘”的系统
- 再讨论受控实盘

## 运行环境基线
当前长期运行基线不是云服务器，而是：
- 一台长期在线的 `Mac mini`
- `launchd` 守护本地服务
- `FastAPI` 托管后端与前端静态产物
- 本地 dashboard 通过浏览器访问

这意味着当前系统是：
- `local-first（本地优先）`
- `Mac-only（仅 macOS 长期运行基线）`
- `paper-first（先模拟盘）`

## 当前产品边界
### 已经成立的主线
- `HK-only（仅港股）` 市场主线
- `dynamic_hk（港股动态选股）`
- `MiniMax` 真链路
- `strategy knowledge layer（策略知识层）` 第一版
- `backtest admission（回测准入）`
- `paper trading（模拟盘）`
- `live readiness（实盘就绪度）`
- `audit trail（审计链）`
- `runtime status（运行状态）`

### 当前不做
- 自动实盘
- 新闻/公告多输入源扩张
- 重型桌面壳
- 云端分布式部署

## 市场与资金假设
当前默认研究和运行前提：
- 市场：港股
- 账户货币：`HKD（港币）`
- 默认模拟盘资金：`100,000 HKD`
- 自动选股需要通过流动性和可负担性过滤

这样做的原因：
- 更接近未来真实小资金实盘规模
- 避免研究出“理论可行但一手都买不起”的候选

当前主运行标的策略：
- 系统允许 `dynamic_hk（港股动态选股）`
- 但当前长期验证主线优先接受高流动性的 ETF（交易所买卖基金）作为稳定运行基准
- `2800.HK（盈富基金）` 仍然是当前的重要基准标的

这样做的原因：
- ETF（交易所买卖基金）更适合作为 `paper trading（模拟盘）` 阶段的低噪声验证对象
- 普通个股的事件风险更高，不适合当前阶段作为唯一主运行基准
- 当前阶段优先验证系统稳定性，而不是放大个股弹性

## 港股数据源与 ETF（交易所买卖基金）实时数据边界
当前港股数据源按用途分层：

### 历史日线主链
- `minshare（港股授权历史日线）`
- `tencent（腾讯）`
- `akshare`
- `yfinance`
- `stooq`

这条链服务于：
- `universe selection（选股）`
- `backtest admission（回测准入）`
- `paper trading（模拟盘）` 信号历史
- `latest quote fallback（最新报价回退）`

### 最新报价主链
- 优先尝试 `minshare（港股实时日线）`
- 若失败，则回退到历史日线最近收盘价

当前一个明确边界：
- `2800.HK（盈富基金）` 属于 ETF（交易所买卖基金）
- 在当前官方权限口径下，它不一定被普通 `港股实时日线` 覆盖
- 因此对 `2800.HK`，系统允许：
  - 实时接口失败
  - 回退到历史日线最近收盘价

当前产品决策：
- 不为了匹配普通 `港股实时日线` 就立刻把主运行标的改成普通个股
- 继续保留 ETF（交易所买卖基金）主线
- 由 `历史日线 + 免费报价源` 作为当前实时兜底
- 若后续确实需要 ETF（交易所买卖基金）更及时估值，再单独补对应实时权限

## 从策略生成到进入模拟盘的真实流程
当前主流程是：
1. `Universe Selection（选股）`
2. `Market Analysis（市场分析）`
3. `Strategy Knowledge（策略知识层）`
4. `Strategy Generation（策略生成）`
5. `Research Debate（研究辩论）`
6. `Backtest Admission（回测准入）`
7. `Governance Decision（治理决策）`
8. `Paper Trading（模拟盘）`
9. `Live Readiness（实盘就绪度）`

关键原则：
- 策略知识层先于策略生成
- 系统先读取知识家族、市场适配和失效模式，再生成当前 proposal
- 回测发生在模拟盘之前
- 没过回测准入的策略，不允许直接进入模拟盘
- 模拟盘不是初筛工具，而是更接近真实运行环境的第二层验证

## 策略知识层的边界
第一版 `strategy knowledge layer（策略知识层）` 的目标不是做策略百科，而是把仓库里已经在使用的策略常识显式化、可审计化。

当前知识来源固定为：
1. 现有 `baseline（基线策略）` 的反向抽象
2. 现有 `market profile（市场画像）` 中已经隐含的市场适配知识
3. 现有 `governance rules（治理规则）` 中已经隐含的失效条件与参数约束

当前仅包含 5 个知识家族：
- `trend_following（趋势跟随）`
- `mean_reversion（均值回归）`
- `breakout（突破）`
- `momentum_filter（动量过滤）`
- `volatility_filter（波动率过滤）`

当前明确不做：
- 不直接导入外部文章
- 不让外部网页原文直接进入 prompt
- 不让未审核知识直接参与门禁
- 不把知识条目当成可执行策略

策略知识层当前承担的职责：
- 解释 baseline（基线策略）属于哪类方法
- 判断 proposal（策略提案）是不是只是轻微换皮
- 判断 proposal 是否踩中某类方法的常见失效模式
- 为 `research debate（研究辩论）`、`risk gate（风险门禁）`、`audit（审计）` 提供共同语言

对应的知识层审计事件：
- `strategy_knowledge_applied（策略知识已应用）`
- `strategy_novelty_reviewed（策略新颖性已评审）`

## 候选策略生命周期
候选策略会落库保留历史证据，但不会无限堆积在主视图里。

当前策略生命周期分层：
- 活动层：`active / recent candidates / current challengers`
- 归档层：历史候选、已淘汰候选、超出观察窗口的候选
- 审计层：保留完整证据链，供历史追溯

当前明确原则：
- 不轻易物理删除历史 proposal
- 但默认视图必须收敛，不能让历史噪音淹没当前判断

## `LLM provider（模型服务）` 的意义
`provider` 字段不是纯展示字段，而是审计字段和分析字段。

它的作用：
- 标识候选策略是由哪个模型服务产出
- 在模型迁移时保留历史分界点
- 区分真实 provider 结果与 `mock fallback（样例回退）`
- 支持 provider cohort 对照、晋级率对照、fallback 率对照

当前原则：
- 历史 proposal / decision / audit 不覆写
- provider 切换必须写审计事件
- 新旧 provider 结果只做并行观察和对照，不直接覆盖活跃策略结论

## Dashboard 的定位
当前 dashboard 不是“控制台优先”，而是：
- `status board（状态看板）`
- `evidence board（证据看板）`
- `review board（评审看板）`

原因：
- 当前没有自动实盘
- 用户现在最主要的需求不是控制系统，而是理解系统、追踪系统、评审系统

因此当前信息架构原则是：
- 首页看摘要
- 详情页承接深信息
- 历史和证据下沉到详情页或折叠组

## 语言与表达规范
长期固定规则：
- 用户可见内容优先中文
- 必要英文术语必须在旁边配中文解释
- 不再只展示英文缩写而不给解释

对模型输出的要求：
- prompt 已要求自然语言字段优先使用简体中文
- 如果历史记录仍是英文，前端可以做中文化标题和术语兜底

## 近期运行与优化原则
当前阶段的优化优先级：
1. 不破坏长期运行
2. 不扩大不必要的功能面
3. 优先加厚 `v2` 证据层
4. 优先做信息架构减法，而不是继续堆字段

当前最重要的工作不是“加更多功能”，而是：
- 持续运行
- 积累样本
- 让证据更厚
- 让界面更易理解

## 未来方向
### `v2`
重点是：
- provider migration governance（模型迁移治理）
- provider cohort history（模型分组历史）
- readiness / paper / runtime 的长期趋势
- 本地长期运行稳定性

### `v3`
在满足准入标准之后，再进入：
- 小资金
- 强风控
- 可回滚
- 可审计

的受控实盘阶段
