# 关键决策

## D-001 项目更名
- 决策：`quant-trader` 更名为 `GobyShrimp`
- 原因：旧名只描述交易执行，新名更适合表达“人类 + LLM 共生协作的策略工厂”

## D-002 B 级自治
- 决策：系统自动推进到模拟盘，不自动进入实盘
- 原因：当前阶段需要让风险管理自动化，但仍避免真实资金暴露

## D-003 LLM 不直接输出执行代码
- 决策：LLM 只能输出 `StrategyDSL`
- 原因：必须保证规则可审计、可验证、可回滚

## D-004 事件输入双轨并存
- 决策：同时保存 `EventStream` 和 `DailyEventDigest`
- 原因：事件流服务审计和 Drill-down，日摘要服务研究与软评分

## D-005 风控门禁优先级
- 决策：三条不可覆盖底线优先于任何 LLM 判断
- 原因：风险约束必须确定性且可解释

## D-006 Dashboard IA 重构
- 决策：以 Agent 指挥台替换旧研究台
- 原因：当前系统核心对象已经从任务列表变为候选池、风险决策、事件摘要和审计链

## D-007 配置边界收敛
- 决策：配置分为 `.env/.env.local`、YAML 默认值、runtime override、数据库数据对象 四层
- 原因：密钥、系统默认行为、运行时切换和研究结果必须分层，否则无法保证安全性、可复现性和可审计性

## D-008 删除无消费配置
- 决策：删除当前没有任何运行路径读取的配置项，而不是继续保留占位字段
- 原因：无消费配置会误导使用者，以为这些字段仍然生效

## D-009 删除旧执行链路
- 决策：删除旧 MCP / DecisionGraph / Orchestrator / Scheduler / policy.yaml / notifier 相关链路，不保留归档和兼容层
- 原因：项目已切到 GobyShrimp 主线，继续保留旧执行叙事只会增加理解噪音和维护风险

## D-010 删除旁路执行模块
- 决策：删除未接入当前 API/Dashboard 主线的 `broker` / `storage` / `reconciler` 模块，以及 OpenAI/Longbridge 兼容配置
- 原因：这些模块只剩测试或历史兼容用途，继续保留会误导读者以为它们仍是当前产品路径

## D-011 事件输入收敛为宏观单通道
- 决策：删除新闻和公告输入，保留宏观输入作为中低频研究的辅助背景
- 原因：当前系统不是高频交易，新闻和公告增加的复杂度高于实际收益；新增输入源应先判断是否是当前策略频率的主驱动

## D-012 风控治理 v1.5
- 决策：把风险判断从“单次分数结论”提升为“治理规则集合”，显式包含晋级阈值、挑战差值、冷却期、暂停阈值和回滚阈值
- 原因：只有把规则显式化并进入证据包、审计记录和 dashboard，系统才能回答“为什么晋级、为什么没晋级、为什么暂停、为什么回滚”

## D-013 实盘准入必须独立于模拟盘晋级
- 决策：模拟盘晋级不等于实盘准入，进入实盘必须满足独立的长期质量、运行稳定性、运营验收和人工确认条件
- 原因：paper 只能证明系统具备受控运行能力，不能单独证明它已经值得承担真实资金风险

## D-014 LLM Provider 迁移优先于桌面壳包装
- 决策：`v2` 先做 provider migration governance，再做 desktop packaging proposal
- 原因：先把“未来换模型不失控”的边界定清楚，再包装产品形态，避免桌面壳先落地却把核心治理问题推后

## D-015 桌面应用只做本地控制面
- 决策：未来的 `GobyShrimp Desktop` 只负责本地服务启动、停止、重启、状态和日志，不搬迁策略生成、风险决策和 paper 账本逻辑
- 原因：当前 FastAPI + Vue + 本地数据库主线已经成立，桌面壳应该是轻包装，而不是第二套核心执行系统

## D-016 中文优先与术语配对
- 决策：用户可见内容默认优先中文；必要英文术语必须同时给出中文解释
- 原因：当前系统术语密度高，若只保留英文会显著提高理解门槛；中文优先更符合当前用户和产品阶段

## D-017 Mac mini 本地长期运行优先
- 决策：当前生产式运行基线定义为 `Mac mini + launchd + 本地 FastAPI 服务 + 浏览器 dashboard`
- 原因：当前阶段重点是持续积累模拟盘、readiness 和 provider cohort 证据，不需要过早引入云部署和分布式复杂度

## D-018 Dashboard 定位为评审看板
- 决策：dashboard 优先按 `状态看板 / 证据看板 / 评审看板` 设计，而不是按强操作型控制台设计
- 原因：当前系统尚未进入自动实盘阶段，用户核心需求是理解、追踪和评审，而不是频繁控制和干预

## D-019 第一版策略知识层先内置、后外接
- 决策：第一版 `strategy knowledge layer（策略知识层）` 只使用仓库现有 baseline、市场画像和治理规则反向抽象出的内置知识目录，不直接导入外部资料
- 原因：先把知识模型、baseline 映射、治理接入和审计语义做稳，再考虑第二阶段的外部资料提炼与审核流程；这样能避免一开始就引入来源不清、质量不稳和不可审计的知识输入

## D-020 多标的扩展优先走“并行研究 + 有限席位”，不走“一条策略覆盖多票”
- 决策：如果后续从当前单标的主线扩展，不要求一条基于某只股票生成的策略去覆盖很多股票，而是优先考虑 `multi-symbol parallel research（多标的并行研究）`、`limited paper slots（有限模拟盘席位）` 和 `portfolio-level allocation（组合层仓位分配）`
- 原因：不同股票的行为结构、风险来源和适配策略差异明显；强行让单标的策略泛化到多票，会削弱解释力、审计能力和治理质量。更合理的扩展方式是每只股票各自生成策略、各自回测、各自评审，再由组合层统一分配资金
